Getting started with humdrumR

Nathaniel Condit-Schultz

2019-12-30

Welcome to “Getting started with humdrumR”! This document will introduce you the basics of humdrumR, so that you can start doing simple analyses of humdrum data. (If you aren’t sure what humdrumR is or what it can do, checkout the Overview of humdrumR before trying to get started.) This tutorial is aimed at novice or even non-programmers—if you are already a strong coder and want to move things a long a little faster, checkout the humdrumR for coders tutorial instead.

This vignette closely parallels information in humdrumR’s code documentation; within an R session, you can find more concise versions of the material in this document by calling:

library(humdrumR)
Loading required package: magrittr

?humdrumR

Quick Start

Let’s just dive right in! First, make sure humdrumR is installed: how to install humdrumR.
Open an R session and load the humdrumR library using the command library(humdrumR)—now you are ready to rock!

HumdrumR comes packaged with a small number of humdrum data files just for you to play around with. These files are stored in the directory where your computer installed humdrumR, in a subfolder called “HumdrumData”. You can move your R session to this folder using R’s “set working directory command”: setwd(humdrumRroot). Once you’re in the humdrumR directory, you can use the base R dir function to see what humdrum data is available to you.

library(humdrumR)

setwd(humdrumRroot)

dir('HumdrumData')
[1] "BachChorales"        "BeethovenVariations" "MozartVariations"   
[4] "RapFlow"             "RollingStoneCorpus"

It looks like there are five folders of humdrum data available to you. Using dir again, we can look inside one: let’s start with the ‘BachChorales’ folder.


dir('HumdrumData/BachChorales')
 [1] "chor001.krn" "chor002.krn" "chor003.krn" "chor004.krn" "chor005.krn"
 [6] "chor006.krn" "chor007.krn" "chor008.krn" "chor009.krn" "chor010.krn"

There are ten files in the folder, named “chor001.krn”, “chor002.krn”, etc. These are simple plain humdrum text files, using the ’**kern’ interpretation to represent ten chorales by J.S. Bach. Take a minute to find the files in your computer’s finder/exporer and open them up with a simple text editor. One of the core philosophies of humdrumR is that we maintain a direct, transparent relationship with our symbolic data—so always take the time to look at your data! You can also do this within Rstudio’s “Files” pane—in fact, Rstudio will make things extra easy for you because you can (within the Files pane) click “More” > “Go To Working Directory” to quickly find the files.

Reading humdrum data

Now that we’ve found some humdrum data to look at, let’s read it into humdrumR. We can do this useing humdrumR’s readHumdrum command. Try this:


readHumdrum('HumdrumData/BachChorales/chor001.krn') -> chor1
Finding and reading files...
    REpath-pattern 'HumdrumData/BachChorales/chor001.krn' matches 1 text files in 1 directory.
One files read from disk. 
Validating one files...all valid.
Parsing one files...Assembling corpus...Done!

This command does two things:

  1. The readHumdrum function will read the “chor001.krn” file into R and create a humdrumR data object from it.
  2. This new object will be saved to a variable called chor1. (The name ‘chor1’ is just a name I chose—you are welcome to give it a different name if you want.)

Cool! Still, looking at a single humdrum file is not really that exciting. The whole point of using computers is working with large ammounts of data. Luckily, humdrumR makes this very easy. Check out this next command:

Notice that all I’ve changed is that instead of writing 'chor001.krn', I wrote 'chor0'. When we feed the string 'chor0' to readHumdrum, it won’t just look for a file called “chor0”; it will read any file in that folder whose name contains the substring “chor0”—which in this case is all ten files! Wow! We’ve now got a “humdrumR corpus of ten files”—and that’s nothing: readHumdrum will work just as well reading hundreds or thousands of files! Notice that when you print humdrumR object, humdrumR shows you the beginning of the first file and the end of the last file, as well as telling you how many files there are in total.

readHumdrum has a number of other cool options which you can read about in more detail in our humdrumR read/write tutorial.

The Data Model

To really make use of humdrumR, you need to understand a bit about how humdrum data is represented in the humdrumR package: the humdrumR “data model.”

The essential thing to understand is how it maps between the humdrum data syntax and a R data.table.

To fully understand this, you should start with at least a basic understanding of the humdrum syntax! Read about the syntax at humdrum.org or check out [our vignette]{HumdrumSyntax.html “The Humdrum Syntax”} on the topic.

data.frames

Data.frames are essential to R. A data.frame is simply a two-dimensional table of named columns. Each column is either a vector or list of values, all of which are the same length.

humdrumR makes use of a popular extension of base-R data.frames, the data.table.

In humdrumR, every single, individual token in a collection of humdrum-syntax text files is given its own row in a data.frame. For example, consider this simple, humdrum-syntax file:

 !!!OTL: Example 1
 **kern                        **kern                        
 *M4/4                         *M4/4                         
 *C:                           *C:                           
 =                             =                             
 2G                            4g                            
 .                             4f                            
 2C                            2e                            
 ==                            ==                            
 *-                            *-                            
NULL

This file contains nineteen individual tokens. To illustrate, here I’ll print the same file, but with each token bracketed by < and >:

<!!!OTL: Example 1>
<**kern>                      <**kern>                      
<*M4/4>                       <*M4/4>                       
<*C:>                         <*C:>                         
<=>                           <=>                           
<2G>                          <4g>                          
<.>                           <4f>                          
<2C>                          <2e>                          
<==>                          <==>                          
<*->                          <*->

Main Fields

The most important fields that are always present in a humdrumR humtable are 1) the “Token” field which contains the original character tokens read from the humdrum data!; and 2) the “Structural” fields which tell us where in the original humdrum data each token came from. The “Structural” fields are what allow humdrumR to (re)construct the original humdrum data—translating between the humdrum-table (data.table) and the plain-text humdrum syntax.

Stops

In humdrum syntax, multiple tokens can be placed “in the same place” (i.e., same record, same spine) by simply separating them with spaces. (This is most commonly used to represent chords in **kern data.) In humdrumR, we call these “Stops”—as always, every humdrum token, including stops, get their own row in a humdrumR humtable. Thus, we need the Stop field to tell us which stop a token came from! In much data, all/most tokens are simply Stop == 1 (the first position), but if there are more than one tokens in the same record/spine, they will be numbered ascending from one:

Doing Stuff!

Ok, we’ve now seen how to read and inspect humdrumR data. We’ve also got some idea of what the humdrum data looks like under the surface: the humdrum table. What can we do with it?

When you create a humdrumR corpus object, using readHumdrum, this object encapsulates a humdrum-table (data.table) object, which is where the data is really stored. We’ve already seen that you can extract that humdrum table directly using getHumtab—and if you are aleady an R pro, you might just want to extract the table an work with it directly. However, what the humdrumR object does is give us some convenient ways of working with that table. For example, we’ve already seen that when we print humdrumR objects, they print out as humdrum syntax, which is easier to read than the complex humdrum table! But the humdrumR does more!

The main thing humdrumR does is allow us to work with the fields of our humdrum-tables in way that is convenient and powerful. First, we create special “do-expressions” that refer to the humtable’s fields. To create these expression, we use the R ~ operator. For instance, I can write things like:

  • ~ Spine + File
  • ~ nchar(Token)
  • ~ paste0(Token, Record)

Notice that all three of these expressions contain the names of humdrumR fields. If you execute these lines in an R session, nothing much will happen: you’ll see that they are saved—‘quoted’—and returned as R formula objects.


~ Spine + File
~Spine + File

~ nchar(Token)
~nchar(Token)

~ paste0(Token, Record)
~paste0(Token, Record)
